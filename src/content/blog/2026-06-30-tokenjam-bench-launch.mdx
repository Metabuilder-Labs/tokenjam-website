<TLDR>
- TokenJam flags which models look safe to downsize. Bench answers the question it can't: does the cheaper model still get the work right, and how much does it actually save?
- Bench runs your task suite on both the original and candidate model, scores pass/fail plus cost on each, and reports the accuracy delta with a Wilson confidence interval and a McNemar test.
- The verdict is per-suite and honest: HOLDS or REGRESSED, measured on the tasks you ran. Never a "certified" guarantee.
- It runs locally. No cloud, no signup. `pipx install tokenjam-bench`, then `tjb run`.
- Bench is the proof layer next to Lens. Lens observes, analyzes, and optimizes. Bench proves the swap before you trust it.

**What is TokenJam Bench?** TokenJam Bench is an open-source tool that measures whether a cheaper model does the same job as the one you run now. You give it a task suite, an original model, and a candidate. It runs both, scores each on pass/fail and cost, and reports where the candidate keeps up and where it breaks.

TokenJam already reads your agent telemetry and flags models that look safe to downsize. That flag is a structural heuristic. It can be wrong. Acting on it without evidence is the part that breaks production quietly, three weeks after you forgot you changed anything. Bench closes that gap.

## The question a cost tool can't answer

Two months ago I wrote that [cost dashboards report the bill and stop short of telling you what to change](/blog/2026-05-31-cost-dashboards-tell-you-the-bill). Reporting spend is summation: walk the spans, multiply by the rate, show the total. Recommending a change is inference. To say "this workload runs fine on Haiku" is a claim about quality on a workload nobody has actually run on Haiku.

Get that claim wrong and the cost is asymmetric. The user applies the swap, their agent degrades in some subtle way, and they lose trust in the tool for good. The downside of a bad recommendation dwarfs the upside of a vague one. That asymmetry is why most cost tools stop at the bill.

The honest fix is evidence pulled from the work itself. Not "Haiku is good at coding," which is a leaderboard claim. Instead: "we ran a code suite on Haiku, it passed 18 of 40 where Opus passed 33 of 40, a 37-point drop the McNemar test puts at p=0.0001." That second sentence is what Bench produces.

## How it works

```
benchmark tasks → run on ORIGINAL model  → score (pass/fail) + cost
                → run on CANDIDATE model → score (pass/fail) + cost
                → report: Δaccuracy (objective) + Δcost, with statistics
```

Bench takes a set of executable tasks where a grader checks the result instead of a human opinion. Start with humaneval or gsm8k, or point it at your own suite. It runs every task twice, once per model. Each run yields a pass or a fail and a measured cost. Then it compares the two.

The comparison is where the statistics earn their place. Bench puts a Wilson confidence interval on each pass rate and runs a McNemar test on the paired results, so a real regression reads differently from sampling jitter. Here is a live run, Opus 4.8 against Haiku 4.5, 40 tasks per suite:

```
OPUS 4.8 → HAIKU 4.5   ·   N = 40 / suite   ·   measured, never "certified"

humaneval (code)   33/40 → 18/40   −37.5pp   McNemar p=0.0001   REGRESSED
gsm8k (math)       40/40 → 39/40    −2.5pp   McNemar p=1.000    HOLDS

COST   $0.3405 → $0.0672   (−80.3% measured)   ·   suites held: 1 of 2
```

Same swap, two verdicts. On math the candidate holds: a 2.5-point drop with a McNemar p of 1.000 is noise, not a regression. On code it breaks: 15 tasks that passed on Opus failed on Haiku, none went the other way, and the 37.5-point drop lands at p=0.0001. Cost fell about 80% either way. Routing the math workload to Haiku is defensible. Routing the code workload would have shipped a quiet quality drop, and Bench caught it before a user did.

![Per-suite verdict from a live Opus 4.8 to Haiku 4.5 run: gsm8k holds, humaneval regresses](/blog/images/bench-run-per-suite.png)

Bench does not make the call for you. It hands you the measured result and the confidence around it.

## Honest by construction

Bench reports a measured pass-rate on the tasks you ran. It never prints "certified," "safe," or "guaranteed." Those words promise something a benchmark cannot deliver: that the model will hold on the inputs you didn't test. The verdict stays scoped to the suite, with its interval visible.

This is deliberate. The eval space runs on leaderboards that crush a model down to one number, and every major agent benchmark has been shown reward-hackable to near-perfect scores. A pass-rate without its confidence interval is marketing. Bench shows the interval and the test, and it says so plainly when a result is too close to call.

Cost is measured the same careful way. On the math suite above, Haiku emitted about 2.1 times the output tokens Opus did. Cost still fell 70.6%, because Haiku's per-token rate is far lower, but the extra verbosity ate into the per-token saving. Bench reports the cost it measured, that wrinkle included, not a clean rate-card delta.

## Where Bench fits

TokenJam Lens observes your agents, analyzes the telemetry, and surfaces five savings areas: downsize, trim, cache, script, and reuse. It points at spend you could recover. Bench is the step after. It proves a specific swap holds before you make it. Lens finds the candidate. Bench tests it. You decide with the evidence on the table.

## More than a two-model diff

The pass-rate proof is the core. Bench does more around it:

- **A local dashboard** (`tjb serve`): offline and self-contained, with an overview, leaderboards, a regression center, and the scenario library on separate screens.
- **Replay mode**: point Bench at your real TokenJam telemetry and test a candidate against the work you actually ran, not just synthetic suites.
- **Agent benchmarks**: multi-turn trajectories with a tool-use safety gate, so you can prove an agent holds, not just a single answer.
- **History and a version-regression matrix**: track a model's pass-rate across TokenJam versions and over time, and flag regressions automatically.
- **Workflow suites**: production-shaped tasks like customer support, RAG, email, and research, beyond humaneval and gsm8k.

![The TokenJam Bench dashboard: leaderboards and a regression center over your recorded runs](/blog/images/bench-dashboard.png)

Everything above runs locally and offline, same as the core proof.

## Try it

Bench runs entirely on your machine. No cloud, no signup.

```bash
pipx install tokenjam-bench
tjb run --original anthropic:claude-opus-4-8 \
        --candidate anthropic:claude-haiku-4-5 \
        --suite humaneval
```
